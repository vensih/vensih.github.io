About the project

Keynest is a minimal, offline desktop password manager. No cloud sync. No accounts. No telemetry. Every credential you store is encrypted on disk and never transmitted anywhere.

The goal was a native experience built on real platform APIs rather than a browser runtime. WinUI 3 handles the UI and a platform-agnostic C# core handles all crypto and storage, keeping the two layers cleanly separated.

This is not a vault for secrets that cannot afford to be wrong. It was built for the mundane stuff: the streaming service password, the Wi-Fi code on a sticky note, the API token nobody rotates. The goal was to keep that kind of thing off plaintext files without adding friction.

What it does

Every feature considered

Offline First

No network dependency whatsoever. Everything runs locally. No accounts to create, no sync service to trust, and no telemetry leaving your machine.

Encrypted at Rest

Entries are encrypted using a C# implementation of the Fernet format: AES-128-CBC with HMAC-SHA256. The HMAC is verified before decryption on every read.

Browse Without a Password

The vault opens automatically on every launch. No password is required to read entries or copy credentials. The master password only gates add, edit, and delete.

App Lock and Auto-lock

App Lock secures the entire app behind your master password on next launch. The edit gate also locks automatically whenever the app loses focus, with no manual step required.

Export and Import

Export your vault to a plaintext JSON file you control. Import merges entries by UUID so duplicates are skipped automatically. Your data stays yours, no migration ceremony.

Native Windows UI

Built with WinUI 3 and the Windows App SDK 2.0. No Electron, no browser runtime, no web stack underneath. The UI is exactly what the OS renders natively.

Security Model

What protects the data

Fernet encryption

Entries are stored as a Fernet token: AES-128-CBC for confidentiality and HMAC-SHA256 for integrity. The HMAC is always verified before decryption, so a tampered vault file is caught before any data is read. The 32-byte encryption key lives only in vault.key on your machine and never leaves it. If that file is lost, the vault is unrecoverable.

PBKDF2 password hashing

The master password is never stored in plaintext. A random 16-byte salt is generated at vault creation and the password is hashed with PBKDF2-SHA256 at 480,000 iterations. On every unlock attempt, the entered password is re-derived with the stored salt and compared using a constant-time equality check. The master password does not affect encryption at all: it only gates whether write operations are permitted.

Windows Hello vault reset

After five consecutive wrong unlock attempts, the only option is a full vault reset. Before any data is deleted, the app triggers a Windows Hello prompt: PIN, fingerprint, or face. If you cannot pass the OS credential check, the reset is blocked. Passing it permanently deletes vault.key and vault.json. All stored entries are gone and cannot be recovered.

A note on scope

Built for the everyday, not the extreme

Keynest handles everyday credentials well. It is not built to withstand a targeted attack. Keep banking details, private keys, and seed phrases somewhere purpose-built for that threat level. The encryption is genuine and nothing leaves your machine, but the design assumption is accidental exposure, not an adversary.
